Aujourd'hui sur Hermes, la grosse priorité c'était de faire le ménage sur WhatsApp. Le user utilisait plus que Telegram, donc autant dégager l'ancienne config. Résultat, Telegram seul reste actif, c'est propre.

Parallèlement on a galéré sur Signal. Le setup signal-cli c'est jamais simple — et aujourd'hui ça a été particulièrement douloureux. D'abord le classique Java version mismatch : signal-cli 0.14.2 réclame Java 23 mais on a dû repasser sur 0.13.1 + Java 21 pour que ça marche. Ensuite le vrai mur, c'est le réseau : le serveur bloque les connexions sortantes vers l'infrastructure Signal. DNS fonctionnel, connexion locale OK, mais dès qu'on tape vers les serveurs Signal c'est mort. On a tenté le linking via QR code mais c'était le bazar — le QR généré par signal-cli n'était pas affiché correctement dans le terminal (problème d'encodage WSL), et même quand on avait quelque chose à scanner, le lien expire super vite et la convo coupait avant la fin. À un moment on a même confondu le scan QR de l'app Signal elle-même avec celui de signal-cli — il faut générer un `signal-cli link` côté laptop et scanner depuis l'app, pas l'inverse. Au final Signal reste down, c'est le réseau le blocker.

Sur un autre front, SearXNG tourne nickel maintenant — port 8081, plus de CAPTCHA ni de rate limiting pour les recherches web. C'est un win positif qui permet de se passer de Google blocks. En discutant, on a évoqué le besoin d'une clé API OpenRouter dédiée pour Hermes et l'accès à un GPU pour des LLMs plus costauds (les modèles 405B c'est pas le même combat sur CPU).

Ensuite au restart, le monitor a planté sur un `[Tool calls: read_f` au démarrage, comme un read_filesystem call qui n'aboutit pas. Faut creuser au prochain lancement. Sinon on a basculé le modèle vers `minimax-m2.7:cloud` via ollama-launch pour tester sans Docker.

C'était un bug qui existait avant nos changements, rien à voir avec les modifs récentes. Solution : retourner à l'original et corriger ce regex.
